Principios Fundamentales

Tres pilares que guían todo el sistema:

1. Gobernanza Explícita

Todos los componentes implementan 8 contratos (ARIA, teclado, WCAG 2.2 AA, variantes, estados). No hay ambigüedad. Consulta AGENT-CONTRACT.md.

2. White Label Operativo

3 marcas (promptea, nova, ocean) × 2 temas (dark, light) reutilizables con data-brand y data-theme. Tokens en component-manifest.json.

3. Descubrimiento Automático

Agentes encuentran todo vía llms.txt → manifest.json → AGENT-CONTRACT.md. Operabilidad es auditada, no advinada.

Design Tokens Platform v2.3.0 — Multi-Platform + 4 Agents + CI/CD

Basado en la tesis del Design.MD: un buen documento Markdown es en sí mismo una pieza de diseño. Este sistema es white label (3 marcas × 2 temas re-tematizables con data-brand/data-theme) y una API de diseño gobernada con PHASES 0-3 completadas:

✅ PHASE 1: Schema
160 tokens con machine-readable metadata (W3C DTCG). Agent 1 valida.
✅ PHASE 2: Multi-Platform
Style Dictionary: Web, Tailwind, iOS, Android, Storybook. npm run tokens:build.
✅ PHASE 3: Agents + CI/CD
4 Agents + governance loops. GitHub Actions pipeline. PR automation.
🤖
Para agentes: ruta obligatoria — llms.txt → component-manifest.json → AGENT-CONTRACT.md. NUEVO: 4 agentes automáticos en scripts/agents/ validan tokens, proponen génesis, detectan hardcoded values, generan changelog. CI/CD en .github/workflows/validate-tokens.yml. Si algo no existe en el manifiesto: escalar, nunca improvisar.

Quick Start — tres rutas de arranque

Comienza según tu rol. El sistema es operado por humanos y agentes IA bajo contratos explícitos (AGENT-CONTRACT.md).

👤 Humanos (2 min)

git clone https://github.com/HAROLDRAGE/ds-base-kit
cd ds-base-kit
open index.html

Edita tokens solo en 05-agentes/component-manifest.json: todo lo demás se derive automáticamente.

Botones y Button Group átomo

Acción interactiva que dispara una operación. Un solo primario por vista.

✔ Do
  • Un solo botón primario por vista
  • Verbo de acción como etiqueta («Guardar cambios»)
✖ Don't
  • Varios primarios compitiendo por atención
  • Etiquetas vagas como «OK» o «Sí»
Contrato ARIA
Elemento nativo <button>: rol, foco y teclado gratis. aria-label en botones solo-icono. aria-busy durante loading.
Contrato de teclado
Tab enfoca · Enter / Espacio activan

Alerts molécula

Mensaje contextual persistente que comunica estado: éxito, información, advertencia o error.

✔ Éxito: cambios guardados correctamente.
ℹ Info: la sincronización tarda unos minutos.
⚠ Advertencia: tu sesión expira pronto.
✖ Error: no se pudo procesar el pago.
✔ Do
  • Icono + palabra + color (triple codificación)
  • role=alert solo para errores críticos
✖ Don't
  • Comunicar el estado solo con color
  • Usar role=alert para mensajes informativos
Contrato ARIA
role="alert" para errores críticos (interrumpe); role="status" / aria-live=polite para éxito y estado.
Contrato de teclado
No interactivo (el botón de cierre, si existe, es un button estándar)
WCAG 2.2
1.4.1 Uso del color · 4.1.3 Mensajes de estado

Dropdown molécula

Menú de acciones contextuales desplegado desde un disparador.

✔ Do
  • Dropdown para acciones; select para elegir un valor
  • Esc cierra y devuelve el foco al disparador
✖ Don't
  • Abrir con hover (inaccesible en táctil y teclado)
  • Menús de más de ~10 acciones sin agrupar

Modal (Dialog) organismo

Diálogo que interrumpe el flujo para una decisión o tarea acotada.

¿Eliminar 3 archivos?

Esta acción no se puede deshacer.

✔ Do
  • Foco al primer control al abrir; regresa al disparador al cerrar
  • Solo para decisiones/tareas acotadas
✖ Don't
  • Modal para contenido que cabe en la página
  • Cerrar sin devolver el foco (usuario «perdido»)
Contrato ARIA
role=dialog + aria-modal + aria-labelledby; focus trap con Tab; al cerrar, el foco vuelve exactamente al elemento que lo abrió.

Toast molécula

Notificación breve y no bloqueante que se anuncia sin robar el foco.

✔ Do

  • Pausar el temporizador al hover/focus
✖ Don't
  • Robar el foco al aparecer

Tooltip átomo

Aclaración breve y no esencial visible al hover Y al focus.

✔ Do

  • Contenido complementario, nunca esencial
✖ Don't

Android · Material Design 3

Guía de adaptación para Android, considerando Material Design 3 (Material You), densidades dp, y las capacidades del sistema operativo.

Consideraciones Clave

🎨 Unidades: dp (density-independent pixels)

Android usa dp, no px. Nuestros tokens (en rem/px) se convierten automáticamente respecto a la densidad de pantalla (160 dpi = 1 dp = 1 px).

👆 Touch Targets: 48×48 dp mínimo

Material Design 3 requiere 48×48 dp para botones e interactivos. Nuestro --touch-target-min: 44px se redondea a 48 dp en implementación Android.

📐 Grid: 4dp baseline

Material Design usa grid de 4 dp. Nuestros espacios (4, 8, 12, 16, 20...) alinean perfectamente con este sistema.

🌈 Dynamic Color (Material You)

En Android 12+, considera extractar colores del wallpaper. Nuestros tokens semánticos facilitan esta integración sin romper el diseño.

⌨️ Navegación: Back + Gestos

Android tiene botón físico "Atrás". Los modales/drawers deben cerrar con gesto back (Contrato 07). No se recomienda patrón iOS de "swipe from left".

📱 Notch & Cutout Support

Usa WindowInsets (API 30+) o DisplayCutout. Nuestros --safe-area-inset-* aplican automáticamente.

iOS · Human Interface Guidelines (HIG)

Guía de adaptación para iOS, considerando HIG, puntos (pt), safe areas, Dynamic Island y características del ecosistema Apple.

Consideraciones Clave

🎨 Unidades: puntos (pt)

iOS usa puntos (pt), no píxeles. 1 pt = 1 px en iPhone 6/7/8/SE, pero 3 px en iPhone 12 Pro Max (3x density). Nuestros rem se convierten automáticamente.

👆 Touch Targets: 44×44 puntos

Apple Human Interface Guidelines requieren 44×44 pt para botones e interactivos. Nuestro --touch-target-min: 44px mapea directamente aquí.

📱 Safe Area & Notch

Dynamic Island (iPhone 15+) y notch requieren safeAreaLayoutGuide. Usa nuestros --safe-area-inset-* o padding(edges: .all) en SwiftUI.

🎯 Vibrancy & Blur (iOS 15+)

iOS favorece efectos de vidrio/vibrancy sobre sombras flat. Nuestros tokens de sombra se adaptan, pero considera UIBlurEffect en overlays.

⌨️ Navegación: Gestos + Back

iOS enfatiza "swipe from left" para retroceder. Modales se cierran con swipe down o botón. No hay botón "Atrás" de sistema (Contrato 05).

🔔 Notificaciones & Haptics

Aprovecha UIImpactFeedbackGenerator para cambios de estado. Nuestros --motion-* se sincronizan con duraciones hapticas estándar.

Referencia: Mapeo de Tokens iOS

Token Nuestro valor iOS HIG
--space-4 16px 16 pt
--radius-md 8px 10-12 pt (HIG)
--touch-target-min 44px 44 pt (Apple estándar)
--shadow-* 5 niveles UIBlurEffect + vibrancy

Web · Consideraciones HTML/CSS/JS

Guía de implementación web, incluyendo responsive design, accesibilidad (WCAG 2.2 AA), SEO y rendimiento.

Markup & Viewport

Meta viewport obligatorio:

<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">

El viewport-fit=cover permite contenido bajo safe areas en iOS (usa padding-safe para el contenido).

Mobile-First Responsive

1️⃣ Estilos base para móvil

Define estilos para 320px primero. Luego agrega media queries para tablets (768px) y desktops (1024px).

2️⃣ Usa CSS Grid & Flexbox

Evita floats. Usa display: grid y grid-template-columns: repeat(auto-fit, minmax(...)) para fluidez automática.

3️⃣ Fluido: rem sobre px

Define tamaños en rem. Escalan con font-size del root. Nuestros tokens usan rem.

4️⃣ Imágenes responsivas

Usa <picture> + <source media="">, o srcset. Carga versiones @2x para retina displays.

5️⃣ Performance: lazy-load

Imágenes: loading="lazy". Iframes: loading="lazy". Reduce CLS/LCP.

6️⃣ Accesibilidad (WCAG 2.2 AA)

Siempre: ARIA, teclado, contraste 4.5:1, sin trampas de foco, skip links, labels explícitos.

CSS Media Queries (Nuestros Breakpoints)

/* Móvil: 320px - 479px */
@media (max-width: 479px) { /* estilos móvil */ }

/* Tablet: 480px - 767px */
@media (min-width: 480px) and (max-width: 767px) { /* estilos tablet */ }

/* Desktop: 768px+ */
@media (min-width: 768px) { /* estilos desktop */ }

/* Large desktop: 1024px+ */
@media (min-width: 1024px) { /* 3+ columnas */ }

Rendimiento & Core Web Vitals

LCP (Largest Contentful Paint)

Meta: < 2.5s. Optimiza hero image, fonts, CSS crítico.

FID / INP (Interactivity)

Meta: < 100ms. Evita JavaScript bloqueante. Usa Web Workers para computación pesada.

CLS (Cumulative Layout Shift)

Meta: < 0.1. Asigna dimensions a imágenes. Evita inserciones dinámicas que empujen contenido.
