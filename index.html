Un sistema de diseño operable por humanos y agentes

Basado en la tesis del Design.MD: un buen documento Markdown es en sí mismo una pieza de diseño. Este sistema es white label (3 marcas × 2 temas re-tematizables con data-brand/data-theme) y una API de diseño gobernada: descubrible (llms.txt), validable (manifest.schema.json) y auditable (Contrato 08 de salida).

🤖
Para agentes: ruta obligatoria de lectura — llms.txt → component-manifest.json → AGENT-CONTRACT.md. Si algo no existe en el manifiesto: escalar, nunca improvisar.

Quick Start — tres rutas de arranque

👤 Humanos (2 min)

git clone https://github.com/promptea/ds-base-kit
cd ds-base-kit
open index.html

Edita tokens solo en 05-agentes/component-manifest.json: todo lo demás se deriva de ahí.

🤖 Agentes externos (3 pasos)

1. GET https://ds-base-kit.tess.page/llms.txt
2. Leer component-manifest.json  → qué existe
3. Operar bajo AGENT-CONTRACT.md → cómo actuar

Si algo no está en el manifiesto: escalar (Contrato 07), nunca improvisar.

Manifiesto IA — component-manifest.json

Qué existe en el sistema, en formato que un agente parsea sin scraping. v1.1.0 — validado contra el schema.

Contratos IA — AGENT-CONTRACT.md v1.1.0

Cómo debe operar el agente. Ahora con Contrato 08 (salida): toda entrega incluye artefacto + doc + diff de tokens + checklist + changelog propuesto + suposiciones. Cada contrato incluye su par Do/Don't ejecutable.

🤝
Regla de oro: si un contrato no puede cumplirse, el agente NO DEBE improvisar — DEBE escalar al humano responsable (Contrato 07).

Skills para agentes — SKILL.md ★ Nuevo

El kit incluye una Agent Skill propia en el formato de Anthropic (carpeta + SKILL.md con frontmatter YAML), instalable en Claude Code con un cp -r. Además, el sistema se integra con skills y estándares que ya existen en el ecosistema.

Gobernanza: toda skill opera bajo su rol de ROLES.md — una skill generadora nunca aprueba su propio trabajo.

Validación — manifest.schema.json

JSON Schema 2020-12: campos obligatorios (states, a11y, when_not_to_use, atomic_level…). Base del "linting de diseño": el agente supervisor valida cada PR automáticamente. ✓ Manifiesto v1.1.0 conforme

Descubrimiento — llms.txt

Estándar emergente: un agente externo (Claude, GPT, Copilot) descubre toda la documentación del sistema con una sola URL, sin parsear HTML.

Matriz de roles de agentes — ROLES.md

Gobernanza para máquinas: permisos separados, nadie aprueba su propio trabajo.

RolPuedeNo puedeContratos clave
GeneradorComponer UI con componentes y tokens existentesCrear tokens o componentes nuevos00–05, 08
SupervisorValidar contra schema, contraste, estados, checklistModificar cualquier artefacto02–04
DocumentadorGenerar/actualizar .md desde código y manifiestoCambiar valores de tokens o componentes06, 08
CuradorProponer extensiones al manifiesto (PR versionado)Aprobar sus propias propuestas07

Flujo: Generador entrega → Supervisor valida → Documentador registra. Toda propuesta del Curador la aprueba un humano.

Interoperabilidad — tokens.dtcg.json

Tokens en formato W3C DTCG ($value/$type): conectan directo con Style Dictionary, Figma Variables y Tokens Studio, no solo con nuestro manifiesto.
